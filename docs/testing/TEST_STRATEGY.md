# 日值测试策略

## 1. 测试目标

日值优先保护三类高风险规则：

- 钱不能算错：账户余额、欠款、流水、还款和转账要能正确回滚。
- 数据不能误删：资产、账户、分类、交易等危险操作必须有确认或阻断。
- 关键联动不能断：资产、附加项、记账、资金账户之间的联动要保持一致。

## 2. 测试分层

### 单元测试 / 服务测试

适合验证不依赖页面点击的业务规则。

优先覆盖：

- 资产成本、附加项成本、转让收入的计算规则。
- 记账新增、编辑、删除后账户余额和资金流水的变化。
- 有资金流水的账户不能删除。
- 有记账记录、资产或账户占用的分类不能删除。
- 分类迁移后交易分类和快照同步更新。
- 备份导入导出的数据校验。

运行命令：

```bash
npm.cmd test
```

### E2E 测试

适合验证真实页面交互和弹窗链路。

优先覆盖：

- 删除前是否弹确认。
- 阻断删除时是否给出提示，且不会误删数据。
- 保存成功后是否误弹未保存提醒。
- 多层弹窗时最上层弹窗是否可点击。
- 图片移除是否需要确认。
- 关键页面是否能完成真实用户操作。

运行命令：

```bash
npm.cmd run test:e2e
```

## 3. 当前已覆盖

### 单元测试已覆盖

当前单元测试集中在服务层和数据层，已覆盖：

- 分类创建、编辑、删除和阻断规则。
- 分类迁移交易后的分类快照更新。
- 资产购买、附加项、转让、删除等联动规则。
- 交易删除后的账户余额和资金流水回滚。
- 有流水账户不能删除。
- 备份导入导出校验。

### E2E 已覆盖

当前 E2E 位于 `e2e/modal-safety.spec.ts`，已覆盖：

- 资产保存后不误弹未保存提醒。
- 附加项保存后不误弹未保存提醒。
- 资产图片移除必须确认。
- 附加项图片移除必须确认。
- 多层弹窗下确认弹窗必须可点击。
- 资产删除必须确认。
- 记账交易删除必须确认。
- 账户删除必须确认。
- 有流水账户不能删除。
- 有记账记录占用的分类不能删除。

## 4. 后续新增规则怎么判断

### 应该写单元测试的情况

如果规则主要影响数据，就优先写单元测试。

例如：

- 金额怎么计算。
- 账户余额怎么变化。
- 删除后哪些记录要保留或回滚。
- 某种状态下是否允许操作。
- 分类、资产、账户之间的关联如何同步。

### 应该写 E2E 的情况

如果规则主要影响用户操作，就优先写 E2E。

例如：

- 按钮点击后是否出现弹窗。
- 表单校验是否显示在页面上。
- 危险操作是否必须二次确认。
- 弹窗叠加时是否能正常点击。
- 保存后页面状态是否正确关闭或刷新。

### 两者都要写的情况

如果一个功能同时影响钱和页面操作，两者都写。

例如：

- 删除交易：单元测试验证账户余额回滚，E2E 验证删除前必须确认。
- 删除账户：单元测试验证有流水不能删，E2E 验证页面提示和阻断。
- 分类迁移：单元测试验证交易数据被迁移，E2E 验证用户操作流程。

## 5. 暂不优先测试

这些内容可以先不投入太多测试：

- 纯视觉样式细节。
- 非关键动画。
- 临时 mock 文案。
- 还没有稳定下来的页面布局。
- 不影响钱、资产、账户和历史记录的轻量交互。

## 6. 提交前建议跑哪些命令

普通业务改动：

```bash
npm.cmd test
npm.cmd run typecheck
```

涉及页面弹窗、删除、保存、上传图片：

```bash
npm.cmd run test:e2e
npm.cmd test
npm.cmd run typecheck
```

准备阶段性打包或交付：

```bash
npm.cmd run test:e2e
npm.cmd test
npm.cmd run typecheck
npm.cmd run build
```
