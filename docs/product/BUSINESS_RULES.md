# 日值业务规则清单

这份文档用于约束前端、IndexedDB、本地服务层、未来后端 API 和测试用例的行为。凡是涉及“钱、资产、账户、分类、删除”的功能，优先以这里的规则为准。

## 0. 实现状态标记

- `[当前已实现]`：当前 Vue + IndexedDB 版本已经按此规则实现或测试覆盖。
- `[前端兼容]`：当前前端为兼容旧数据或过渡模型保留，未来可能收敛。
- `[后端规划]`：后端/API 阶段再实现，当前前端不一定有完整界面或接口。
- `[待决策]`：业务方向还没最终定死，开发前需要再次确认。

## 1. 核心原则

- 钱不能算错：交易、账户余额、负债欠款、资金流水必须能正确写入和回滚。
- 链路不能断：资产、附加项、记账、资金账户之间的关联必须同步维护。
- 历史不能乱改：历史交易默认保留发生时的分类快照，避免分类改名后旧账含义变化。
- 删除必须慎重：所有删除操作都要二次确认；有历史数据占用的关键对象不能直接删除。
- 业务入口要唯一：系统生成的联动账单，必须回到对应业务入口维护，不能在记账页随便改。

## 2. 资产规则

### 2.1 资产状态

状态：`[当前已实现]`

资产状态使用枚举值，不使用中文文案入库：

- `using`：使用中。可编辑资产档案，可新增/编辑附加项，可转让、标记闲置或标记处置。
- `idle`：闲置。可编辑资产档案，可恢复使用中，可处置；新增/编辑附加项时应提醒用户。
- `transferred`：已转让。资产和附加项只读，只允许撤销转让或删除资产档案。
- `disposed`：已处置。资产和附加项只读，只允许恢复使用中或删除资产档案。

禁止规则：

- 已转让资产不能重复转让。
- 已处置资产不能转让。
- 已转让、已处置资产不能新增或编辑附加项。
- 已转让状态下不显示“恢复使用中 / 标记闲置 / 标记已处置”这类冲突操作。

### 2.2 资产图片

状态：`[当前已实现]`

- 正式模型使用 `attachments` 保存图片、订单、发票、凭证等附件。
- 资产图片附件的 `type` 为 `asset_image`。
- 多图按 `isCover` 和 `sort` 排序展示。
- 没有上传资产图时，页面只显示一张系统占位图，内容取资产名称首字符。
- `imageUrl` / `imageUrls` 只作为兼容输入或表单草稿，不作为正式长期模型字段。

### 2.3 资产成本

状态：`[当前已实现]`

资产表只存事实字段，派生成本不落库：

```text
includedAddonCost = sum(direction = expense 且 includedInCost = true 的附加项金额)
totalCost = originalCost + includedAddonCost
```

规则：

- 原始购入价为 `originalCost`。
- 只有支出型附加项，并且 `includedInCost = true`，才计入资产总成本。
- 收入型附加项不减少资产历史总成本。
- 当前估值 `currentValue` 是独立字段，可手动或由转让产生。
- 已损益、日均成本等展示指标应由资产、附加项、当前估值和使用天数派生计算。

### 2.4 新增资产

状态：`[当前已实现]`

新增资产时，如果选择付款账户：

1. 创建资产。
2. 创建资产购买交易，`type = asset_purchase`，`businessType = asset_purchase`。
3. 创建账户流水。
4. 扣减付款账户余额或增加负债账户欠款。
5. 将交易 id 写回资产的 `purchaseTransactionId`。

如果没有选择付款账户：

- 只创建资产档案。
- 后续补充付款账户时，由资产编辑动作创建对应购买交易和账户流水。

### 2.5 编辑资产

状态：`[当前已实现]`

编辑以下字段时，需要同步资产购买交易、账户余额和账户流水：

- 购入价 `originalCost`
- 购买日期 `purchaseDate`
- 购买商家 `merchant`
- 付款账户 `paymentAccountId`

如果从“有付款账户”改成“无付款账户”：

- 删除对应资产购买交易。
- 回滚原账户余额。
- 删除对应账户流水。
- 清空 `purchaseTransactionId`。

### 2.6 删除资产

状态：`[当前已实现]`

删除资产档案不等于撤销消费。

当前规则：

- 删除资产档案。
- 删除该资产下的附加项档案。
- 删除该资产下的旧部件变动兼容记录。
- 保留历史交易和账户流水。
- 历史交易解除 `assetId`、`addonId`、`partEventId` 关联。

`[待决策]` 如果用户想撤销购买，后续应提供单独的“撤销资产购买”流程，不能混在删除资产里。

## 3. 附加项规则

附加项用于记录资产生命周期内发生的补充支出或回收收入。原“部件变动”主流程并入附加项。

### 3.1 附加项方向

状态：`[当前已实现]`

附加项支持两种方向：

- `expense`：新买配件、维修、保养、升级、耗材等支出。
- `income`：卖掉旧配件、会员共享回款、售后退款、维修赔付、残值回收等收入。

### 3.2 附加项类型

状态：`[当前已实现]`，但收入型展示文案仍属于前端映射。

支出型可展示为：

- 配件
- 维修
- 保养
- 升级
- 耗材
- 其他

收入型可展示为：

- 配件转卖
- 共享回款
- 退款赔偿
- 残值回收
- 耗材回款
- 其他收入

注意：当前数据模型中 `type` 仍复用 `accessory / repair / maintenance / upgrade / consumable / other`，收入型展示文案由前端根据 `direction` 转换。

### 3.3 新增附加项

状态：`[当前已实现]`

新增附加项时：

1. 创建附加项记录。
2. 如果选择账户，创建一条联动交易。
3. 如果 `direction = expense`，交易类型为 `expense`。
4. 如果 `direction = income`，交易类型为 `income`。
5. 创建账户流水并更新账户余额。
6. 只有 `direction = expense` 且 `includedInCost = true` 时，资产总成本才会增加。

### 3.4 编辑附加项

状态：`[当前已实现]`

编辑附加项时：

- 同步更新对应的联动交易。
- 名称、金额、日期、账户、商家、备注变化时，交易和账户流水要同步变化。
- 方向从支出切到收入时，要回滚原支出影响，再写入收入影响。
- 方向从收入切到支出时，要回滚原收入影响，再写入支出影响。
- 附加项类型、备注、图片属于展示属性，但如果类型影响交易分类，也要同步交易分类。

### 3.5 附加项图片

状态：`[当前已实现]`

- 附加项也支持 `attachments`。
- 附加项图片同样使用 `type = asset_image`。
- 附加项详情、列表和大图预览应优先展示附件图片。
- 移除图片必须二次确认。

## 4. 记账规则

### 4.1 记账和转账边界

状态：`[当前已实现]`

产品上拆成两条线：

```text
记账 = 支出 + 收入
转账 = 账户之间的金额流转
```

规则：

- 记账分类只服务 `expense` 和 `income`。
- 转账不放进日常记账分类，也不参与收入/支出统计。
- 转账只影响账户余额和账户流水。
- 还款是特殊业务，和普通生活支出分开处理。

### 4.2 交易类型

状态：`[当前已实现]`

当前模型支持：

- `expense`：支出。
- `income`：收入。
- `transfer`：账户转账。
- `refund`：退款。
- `repayment`：还款。
- `asset_purchase`：资产购买。

`businessType` 用于说明业务来源：

- `normal`：普通记账。
- `asset_purchase`：资产购买。
- `asset_addon`：资产附加项。
- `asset_transfer`：资产转让。
- `debt_repayment`：负债还款。
- `refund`：退款。
- `account_transfer`：账户转账。
- `legacy_part_event`：旧部件变动兼容数据。

### 4.3 系统生成交易保护

状态：`[当前已实现]`

以下交易不能在记账页直接编辑或删除：

- 资产购买交易：`type = asset_purchase` 或 `categoryId = tx-asset-purchase`。
- 资产转让收入：`categoryId = tx-asset-transfer`。
- 附加项交易：存在 `addonId`。
- 旧部件变动交易：存在 `partEventId`。

原因：这些交易同时影响资产、账户和账户流水，必须回到对应业务入口维护。

### 4.4 普通交易

状态：`[当前已实现]`

普通交易新增时：

- 必须有正金额。
- 必须选择可用交易分类。
- 必须选择账户。
- 创建交易后同步创建账户流水，并更新账户余额。

普通交易编辑时：

- 如果金额、类型、账户变化，需要回滚旧账户影响，再写入新账户影响。
- 如果分类变化，需要更新交易分类和分类快照。

普通交易删除时：

- 必须弹窗确认。
- 删除交易。
- 删除对应账户流水。
- 回滚账户余额。

### 4.5 交易分类快照

状态：`[当前已实现]`

交易创建和迁移时保存：

```ts
categorySnapshot: {
  categoryName: string;
  subCategoryName?: string;
}
```

规则：

- 分类改名默认不改历史交易快照。
- 分类迁移会重写历史交易的分类 id 和快照。
- `[待决策]` 未来如果提供“同步历史记录”开关，必须明确提示影响范围。

## 5. 资金账户规则

### 5.1 账户方向

状态：`[当前已实现]`

账户分为两种方向：

- `asset`：资产账户，例如现金、支付宝余额、微信余额、储蓄卡、投资账户。
- `liability`：负债账户，例如信用卡、花呗、借呗、贷款。

### 5.2 账户余额变化

状态：`[当前已实现]`

资产账户：

- 收入、退款：余额增加。
- 支出、资产购买、还款：余额减少。

负债账户：

- 支出、资产购买：欠款增加。
- 收入、退款、还款：欠款减少。

### 5.3 账户转账

状态：`[当前已实现]`

转账规则：

- 转出账户和转入账户不能相同。
- 创建一条 `type = transfer`、`businessType = account_transfer` 的交易。
- 创建两条账户流水：转出一条、转入一条。
- 转账不计入日常收入/支出统计。

### 5.4 还款

状态：`[当前已实现]`

还款规则：

- 还款账户必须是资产账户。
- 被还账户必须是负债账户。
- 还款金额不能大于当前欠款。
- 创建一条 `type = repayment`、`businessType = debt_repayment` 的交易。
- 资产账户余额减少。
- 负债账户欠款减少。
- 删除还款交易时，要同时回滚两个账户。

### 5.5 删除账户

状态：`[当前已实现]`

删除账户规则：

- 必须弹窗确认。
- 不存在资金流水的账户可以删除。
- 已有资金流水的账户不能删除。
- `[后端规划]` 有流水账户后续应提供“停用账户”能力，而不是物理删除。

## 6. 分类规则

### 6.1 分类域

状态：`[当前已实现]`

分类按模块分域：

- `asset`：资产分类。
- `transaction`：记账分类。
- `account`：账户分类。

### 6.2 记账分类层级

状态：`[当前已实现]`

记账分类支持两级：

- 一级分类：例如三餐、出行、工资、外快。
- 子分类：例如早餐、午餐、晚餐；高铁、飞机、地铁；A 公司工资、剪辑外快。

规则：

- 暂不支持三级分类。
- 子分类必须和父分类属于同一模块。
- 子分类必须和父分类同类型，例如支出子分类不能挂到收入父分类下。
- 父分类停用后，不能继续新增子分类。

### 6.3 删除分类

状态：`[当前已实现]`

删除分类必须先判断占用情况。

交易分类：

- 如果该分类或其子分类下有记账记录，不能删除。
- 必须先把对应记账记录改到其他分类，或执行分类迁移。
- 如果没有交易占用，允许删除，连同未占用子分类一起删除。

资产分类：

- 如果已有资产使用该分类，不能删除。
- 必须先把对应资产改到其他分类。

账户分类：

- 如果已有账户使用该分类类型，不能删除。
- 必须先把对应账户改到其他类型。

### 6.4 分类迁移

状态：`[当前已实现]`

分类迁移规则：

- 只允许迁移记账分类。
- 只能迁移到同类型分类，例如支出到支出、收入到收入。
- 目标一级分类不能是子分类。
- 如果指定目标子分类，目标子分类必须属于目标一级分类。
- 迁移后要更新交易的 `categoryId`、`subCategoryId` 和 `categorySnapshot`。

## 7. 旧部件变动兼容规则

状态：`[前端兼容]`

当前产品不再把“部件变动”作为主流程。

新的主流程：

- 新买 CPU、硬盘、内存：创建支出型附加项。
- 卖掉原 CPU：创建收入型附加项。
- 会员分享给别人并收到钱：创建收入型附加项。

兼容规则：

- `assetPartEvents` 仅作为早期本地数据兼容表保留。
- 缺失交易的旧部件变动会在快照读取时尝试补齐交易。
- `[后端规划]` 后端 v1 不建议暴露独立的部件变动 API。

## 8. 备份和恢复规则

状态：`[当前已实现]` + `[后端规划]`

- 当前 MVP 使用 IndexedDB 本地存储。
- 备份导出应包含资产、附加项、旧部件变动、账户、交易、账户流水、分类、设置和元数据。
- 导入前必须校验是否为日值备份格式。
- `[后端规划]` 恢复备份会替换本地表数据，未来界面上必须明确提示风险。

## 9. UI 安全规则

状态：`[当前已实现]`

- 所有删除操作都必须使用自定义确认弹窗，不能用浏览器原生 `confirm`。
- 保存成功后不能误弹“未保存提醒”。
- 表单缺少必填项或金额非法时，必须在页面上显示错误，不允许只在控制台抛错。
- 多层弹窗时，最上层弹窗必须可点击，确认弹窗层级必须高于业务弹窗。
- 图片移除必须二次确认。
- 日期选择后应自动关闭日期面板，并把日期写回输入框。
- 下拉菜单、更多菜单类轻量菜单优先支持 hover 展开，关键操作仍需点击确认。

## 10. 当前测试映射

服务层测试已覆盖：

- 资产购买创建交易、账户流水和余额变化。
- 附加项支出和收入的账户联动与成本计算。
- 附加项方向切换时的交易和账户回滚。
- 资产购买金额、付款账户编辑后的账户回滚。
- 系统生成交易不能在记账页直接删除。
- 普通交易删除后的账户回滚。
- 账户转账、还款和还款撤销。
- 资产转让和撤销转让。
- 删除资产时保留历史交易但解除关联。
- 已转让或已处置资产禁止写入附加项。
- 分类删除阻断和分类迁移。
- 备份导出、校验和恢复。

E2E 测试已覆盖：

- 资产、附加项保存后不误弹未保存提醒。
- 资产图片和附加项图片移除必须确认。
- 多层弹窗层级可点击。
- 资产删除必须确认。
- 记账页删除交易必须确认。
- 账户删除必须确认，且有流水账户不能删除。
- 分类有记账记录时不能删除。

## 11. 后续待决策

- 是否提供“撤销资产购买”独立流程。
- 已有流水账户的“停用账户”界面和统计口径。
- 后端同步后，历史交易快照是否支持批量重算。
- 多币种账户之间转账是否支持汇率。
- 资产删除后是否保留一个“已删除资产名称快照”，用于历史交易展示。
- 备份恢复是否需要版本迁移脚本和冲突预览。
