# Rizhi 项目设计

## 1. 产品定位

Rizhi 是一个面向 PC 网页端的个人生活资产与现金流管理应用。

它要解决的不是单纯“记一笔账”，而是把下面三件事真正关联起来：

- 我买了什么。
- 我从哪个账户花的钱。
- 这笔钱后来变成了什么资产，以及这个资产后续用了多久、还值多少钱。

核心目标是帮助用户理解：

- 钱花到哪里了。
- 钱沉淀成了哪些物品资产。
- 每个物品实际用了多久。
- 每个物品真实的日均成本是多少。
- 自己当前的资金、负债和净资产状态。

一句话定位：

```text
一个把物品、消费和资金账户真正联动起来的个人生活资产管理工具。
```

## 2. 主导航

第一版建议使用五个主板块：

1. 看板
2. 物品资产
3. 记账
4. 资金
5. 我的

## 3. 核心模块

### 3.1 看板

看板用于展示用户当前生活资产和资金状态的总览。

核心信息：

- 净资产
- 总资产
- 总负债
- 本月收入
- 本月支出
- 本月预算剩余
- 最近购买的物品
- 日均成本最高的物品
- 即将过保的物品
- 即将还款的账户
- 闲置较久的物品

看板不应该只是堆图表，而应该优先展示“需要用户注意的事情”。

### 3.2 物品资产

物品资产用于记录用户购买、使用、闲置、转卖、报废的一切物品。

典型分类：

- 数码设备
- 衣物鞋子
- 家电
- 家具
- 家居用品
- 工具
- 箱包
- 图书
- 运动器材
- 其他个人物品

物品资产字段：

- 名称
- 分类
- 品牌
- 型号
- 购买日期
- 原始购买价
- 付款账户
- 购买渠道
- 保修期限
- 过保日期
- 当前状态
- 预计使用年限
- 当前估值
- 备注
- 图片、发票或订单凭证

资产状态：

- 使用中
- 闲置
- 已转卖
- 已报废
- 已借出
- 已丢失

自动计算内容：

- 持有天数
- 使用天数
- 距离过保天数
- 资产总成本
- 日均成本
- 当前估值亏损
- 转卖后的真实使用成本

### 3.3 资产附加项

物品资产不是一笔固定金额，它可以随着后续购买配件、维修、保养等行为不断变化。

示例：

- 手机原始购买价：2000 元
- 手机壳：15 元
- 钢化膜：20 元
- 充电器：99 元
- 手机资产总成本：2134 元

附加项字段：

- 名称
- 类型
- 金额
- 购买日期
- 付款账户
- 记账分类
- 关联资产
- 是否计入资产成本
- 备注
- 凭证图片

附加项类型：

- 配件
- 维修
- 保养
- 升级
- 耗材
- 服务
- 其他

核心规则：

- 可以在物品资产详情页添加附加项。
- 可以在记账时选择“关联到已有资产”，并创建附加项。
- 如果选择“计入资产成本”，该附加项会影响资产总成本和日均成本。
- 如果只选择“相关支出”，该记录只与资产关联，但不影响资产总成本。
- 修改或删除附加项时，需要同步影响对应记账记录、资金账户和资产计算结果。

推荐计算公式：

```text
资产总成本 = 原始购买价 + 计入成本的附加项 - 转卖收入
日均成本 = 资产总成本 / 持有天数
```

### 3.4 记账

记账用于记录所有资金流动。

交易类型：

- 支出
- 收入
- 转账
- 还款
- 退款
- 借入
- 借出
- 资产购买
- 资产附加项

记账字段：

- 交易类型
- 金额
- 日期和时间
- 分类
- 付款账户或收款账户
- 关联资产
- 关联附加项
- 商家
- 渠道
- 备注
- 图片或凭证

关键规则：

- 买入一个物品资产时，应同时生成一条记账记录和一条物品资产记录。
- 买入资产附加项时，可以选择是否关联到已有资产。
- 账户之间转账不应该计入收入或支出。
- 信用卡、花呗还款不应该重复计算为支出，因为真实消费发生在刷卡或花呗支付时。
- 退款应尽量关联原始交易。

资产购买示例流程：

1. 用户记录购买手机，金额 2000 元，付款账户为支付宝。
2. 系统创建一条支出记账记录。
3. 系统从支付宝账户扣除 2000 元。
4. 系统创建一条手机物品资产。
5. 手机资产开始自动计算持有天数、过保状态、日均成本。

资产附加项示例流程：

1. 用户记录购买手机壳，金额 15 元。
2. 用户选择关联资产为“手机”。
3. 用户选择“计入资产成本”。
4. 系统创建一条支出记账记录。
5. 系统从所选资金账户扣除 15 元。
6. 手机资产总成本从 2000 元变为 2015 元。

### 3.5 资金

资金用于记录所有现金、余额、银行卡、信用账户和负债。

账户类型：

- 现金
- 支付宝余额
- 微信余额
- 储蓄卡
- 信用卡
- 花呗
- 借呗
- 贷款
- 应收款
- 应付款
- 投资账户

账户字段：

- 账户名称
- 账户类型
- 当前余额
- 信用额度
- 已用额度
- 账单日
- 还款日
- 所属机构
- 当前状态
- 备注

核心统计：

- 总现金资产
- 总负债
- 净资产
- 可用信用额度
- 即将还款金额

推荐计算公式：

```text
净资产 = 正向资产总额 - 负债总额
```

资金必须和记账联动：

- 支出会减少现金类账户，或增加信用类负债。
- 收入会增加对应账户余额。
- 转账会减少一个账户，同时增加另一个账户。
- 还款会减少现金账户，同时减少负债账户。
- 退款会增加原付款账户或指定收款账户。

## 5. 关键交互流程

### 5.1 从记账创建资产

1. 用户新增一条支出记录。
2. 用户选择“这是一个资产购买”。
3. 用户填写资产信息。
4. 系统创建记账记录。
5. 系统创建物品资产。
6. 系统更新所选资金账户余额。
7. 看板相关统计更新。

### 5.2 从资产详情创建附加项

1. 用户打开某个资产详情页。
2. 用户点击“添加附加项”。
3. 用户填写金额、日期、账户，并选择是否计入资产成本。
4. 系统创建资产附加项。
5. 系统创建关联记账记录。
6. 系统更新所选资金账户余额。
7. 系统重新计算资产总成本和日均成本。

### 5.3 转卖资产

1. 用户将资产标记为已转卖。
2. 用户填写转卖价格、转卖日期、收款账户。
3. 系统创建一条收入记账记录。
4. 系统增加收款账户余额。
5. 系统将资产状态更新为已转卖。
6. 系统计算转卖后的真实使用成本。

### 5.4 账户之间转账

1. 用户选择交易类型为转账。
2. 用户选择转出账户和转入账户。
3. 系统减少转出账户余额。
4. 系统增加转入账户余额。
5. 该交易不计入收入和支出统计。

### 5.5 信用卡消费与还款

1. 用户记录一笔信用卡支出。
2. 系统增加信用卡已用额度或负债。
3. 用户之后记录还款。
4. 系统减少现金类账户余额。
5. 系统减少信用卡负债。
6. 还款不重复计入支出。

## 8. 设计原则

1. 用关联记录代替重复记录。
2. 转账必须和收入、支出分开。
3. 每一次资金流动都应该影响一个或多个资金账户。
4. 每一次资产购买都应该能追溯到对应记账记录。
5. 资产成本应该允许随着时间变化。
6. 日均成本要清晰可见，但不要给用户造成压力。
7. 看板应该优先展示需要注意的事情。
8. 修改有关联的数据时，保存前应提示会影响哪些内容。

## 9. 待确认问题

- 资产日均成本应该按“持有天数”计算，还是按“实际使用天数”计算？
- 维修费用默认是否计入资产成本，还是每次让用户选择？
- 附加项是否可以单独转卖或报废？
- 删除记账记录时，是否自动反向修正账户余额？
- 第一版是否支持多货币？
- 预算是否进入 MVP，还是放到第二版？
- 数据策略是本地优先、云端优先，还是本地与云端混合？
